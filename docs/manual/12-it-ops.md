## Sites

A **Site** is the durable top-level container for Server Rooms and Racks. It is not itself a Connection and owns no live Session or secret. When IT Ops has no Site data, KKTerm keeps an undeletable Default Site so the topology always has a safe parent container.

Open Sites from the add menu beside `itops.title` and choose `itops.racks.addSite`. The New Site dialog explains that a Site can model a region, location, customer, project, or other infrastructure grouping. Enter `itops.sites.nameLabel`; Connection binding is not part of Site creation. The header's shared icon picker customizes the Site icon. Save with `itops.actions.create`.

The left column includes the `itops.title` label/icon and can be resized like the Connection Tree. Use the title-bar button immediately before `app.aiAssistant` to hide or show it. Site children are predefined navigation destinations, not stored folders. `itops.navigation.serverRooms` preserves the existing Server Room → Rack topology tree; `itops.tabs.hosts`, `itops.tabs.autos`, and `itops.navigation.runHistory` each open their own Site-scoped page. `itops.tasks.heading` is global and therefore appears once outside all Sites.

## Hosts

A **Host** is a durable inventory entry for one device or guest in a Site, addressed by hostname. The device itself can be a Host, and a Host can carry **child Hosts** — its virtual machines or containers (`itops.hosts.kind.physical`, `itops.hosts.kind.vm`, `itops.hosts.kind.container`, `itops.hosts.kind.other`). The Site-owned `itops.tabs.hosts` page lists the selected Site's Hosts as a parent/child tree with a count roll-up (`itops.hosts.hostCount`) and owns Host import and manual Task execution. Its `itops.hosts.taskStatusColumn` column shows queued, running, successful, or failed progress for the active Batch Run; `itops.hosts.lastRunStatusColumn` shows the newest completed result associated with the Host's bound execution Connection.

When the Site has no Hosts, the shared compact empty hint (`itops.hosts.empty`) includes an inline `itops.hosts.importAction` action. IT Ops uses the same no-glyph, one-sentence hint treatment for empty Hosts, Automations, Run History, Task Library, Site topology, Server Rooms, and Racks.

**Import.** `itops.hosts.importAction` opens a dialog (`itops.hosts.importTitle`) that accepts a pasted hostname list — one hostname per line, commas also separate, and `#` starts a comment (`itops.hosts.importHint`). Blank lines and duplicates are skipped; the Status Bar reports the outcome (`itops.hosts.importedNotice`). Imported Hosts are created as physical devices at the top level and are **scanned automatically** for remote-orchestration endpoints.

**Connectivity scan.** The scan probes each Host for SSH (port 22), WinRM (5985/5986), and HTTPS (443, a management-interface hint) and stores the snapshot on the Host — never live Session state and never a secret. Each row shows the detected endpoints as chips, `itops.hosts.scanning` while a probe runs, `itops.hosts.scanPending` for a never-scanned Host, or `itops.hosts.scanNone` when nothing answered. `itops.hosts.rescanAllAction` rescans the whole Site; each row's `itops.hosts.rescanAction` rescans one Host.

**Editing.** `itops.hosts.addAction` and each row's `itops.hosts.editAction` open the Host dialog (`itops.hosts.dialogTitleAdd` / `itops.hosts.dialogTitleEdit`) with `itops.hosts.hostnameLabel`, `itops.hosts.labelLabel`, `itops.hosts.kindLabel`, the `itops.hosts.parentLabel` select (`itops.hosts.parentNone` keeps it top level; a Host can never be its own ancestor), and `itops.hosts.notesLabel`. A row's `itops.hosts.addChildAction` pre-selects that Host as the parent. Deleting a Host asks for confirmation (`itops.hosts.deleteConfirmTitle`); its child Hosts are kept and move up one level, and no Connection is affected.

**Connection bindings.** Each row's link chip opens `itops.hosts.bindingsTitle`, where one Host binds any number of existing Connections — for example its SSH terminal Connection plus an HTTPS URL Connection to its management interface (`itops.hosts.bindingsHint`). Bindings are ordered soft references; deleting a Connection never deletes the Host.

**Rack View callouts.** The Rack Device editor's `itops.racks.boundHostCount` action opens a dedicated Host-binding editor for the Host the device *is*; that device's balloon callout in Rack View then lists the Host's hostname and its child Hosts (`itops.hosts.childOverflow` summarizes the rest), so a rack slot reads as "this machine, and the guests running on it".

## Batch Runs and Run History

A **Batch Run** executes a reusable Task or ad-hoc Script Batch Task across resolved targets. It is an execution event, not an independent page. On the Site-owned Hosts page, select one or more Hosts with an SSH Connection binding and choose `itops.actions.runTask`. The launcher keeps that Host selection fixed, shows it with `itops.batchRuns.scopeSelectedHosts`, and lets you choose a global reusable Task with `itops.batchRuns.taskSourceLabel` or enter a one-off Script with `itops.batchRuns.adHocTask`. Selecting a saved Task shows its Script or Playbook definition in the same editor-style code or node-canvas presentation used by the Task Library, but read-only; the ad-hoc Script remains editable. Choose `itops.actions.run` to start the Batch Run. For each selected Host, resolution uses its first bound SSH Connection; a Host without an SSH binding cannot be selected for a run.

The separate `itops.navigation.runHistory` page shows the selected Site's live run and completed reports. It is an audit surface with no start or rerun action; manual execution begins from Hosts, while Automations can also create runs.

Before any run exists, `itops.batchRuns.historyEmptyHint` explains where Run History comes from without presenting a setup action on this read-only page.

## Task Library

A **Task** is a reusable script or Playbook definition; it does not own targets. Open the global `itops.tasks.heading` destination to create, edit, or delete either kind. `itops.tasks.applicableOsLabel` accepts one or more supported operating systems; `itops.tasks.applicableOs.any` is exclusive with specific platforms. The metadata documents compatibility and filters the library, but does not automatically remove selected Hosts from a run. The spreadsheet-style list shows each Task's type, Applicable OS, execution count, failed-host count, and `itops.tasks.viewHistory` link. The link opens the most recent Site Run History attributed to that stable Task id; older or ad-hoc rows are not guessed from their labels. Manual execution starts from selected Hosts with `itops.actions.runTask`; the launcher then offers definitions from the Task Library. Deleting a Task does not change saved Run History.

The Task Library includes app-owned, inspection-only built-ins for Linux, macOS, Windows, Cisco IOS, Cisco NX-OS, FortiOS, Juniper Junos, and Arista EOS. They cover identity, uptime, resource usage, interfaces, routing and DNS, and recent logs. A built-in row shows `itops.tasks.builtInBadge`; choose `itops.tasks.duplicateBuiltin` to create an editable user copy. Built-ins themselves cannot be edited or deleted and are refreshed when the catalog changes.

The reusable Playbook editor uses an ordered workflow canvas similar to the Automation editor. Choose the `itops.tasks.addNode` plus button on any connector, then choose `itops.tasks.addCommand`, `itops.tasks.addSudo`, or `itops.tasks.addAi` from the popup to insert that node at the connector's exact position. The bottom add-node bar appends the same node types to the end of the chain. Select a node to edit it in the inspector. A Sudo node stores its password in the configured secret vault and keeps only an opaque reference in the Task; credential typing stays isolated from canvas rendering and `itops.tasks.credentialStoredDetail` confirms durable state. At run time KKTerm opens one interactive SSH shell per Host, validates sudo once, and lets the remote sudo timestamp apply to the remaining nodes in that run. The password is not stored in Task JSON or Run History.

Add `itops.tasks.addAi` after a node whose output needs interpretation. The AI node sends the preceding node output to the currently configured AI Assistant provider as untrusted data and applies the instruction in `itops.tasks.aiInstructionLabel`. It accepts only three structured results: `continue` runs the next node, `success` ends that Host successfully, and `fail` stops that Host as failed. Tools are disabled for this evaluation, and model output is never executed as a command. Provider errors or malformed decisions fail that Host. Because the previous output leaves the device for the configured provider, do not place an AI node after a command that prints secrets or other data that must remain local.

Execution fans out with bounded concurrency. The live view shows queued, running, successful, and failed hosts and streams each host's combined output. `itops.actions.cancel` stops an active run. A completed run is written to Recent runs; select an entry to open its read-only Run Report and inspect the saved per-host output.

The script is state-changing operator input. Review it and the target Site before starting. Authentication secrets remain in the OS keychain and are not copied into the Site or report.

## Automations

An **Automation** is a durable trigger → optional condition → ordered actions rule. Its saved definition persists and enabled rules re-arm when KKTerm launches; the live **Watchdog** runtime performs sampling and execution in memory.

Each Automation carries a durable Site binding, chosen with the `itops.editor.siteLabel` select in the editor header (`itops.editor.siteNone` leaves it unbound). The Site-owned `itops.tabs.autos` page lists the Automations bound to the selected Site; an Automation created there is bound to it by default. Rules saved before the binding existed show where they can be inferred: a Batch Run action targeting that Site, or a host-addressed trigger (ping / TCP reachable) watching one of the Site's resolved member hosts. `itops.actions.newAutomation` opens the editor. In the editor, name the rule, configure its trigger and condition, then add typed actions with `itops.actions.addAction`. Available actions include notify, popup, email, webhook, and running a Batch Run. The editor's Test action samples/evaluates the rule and previews actions without sending email, calling a webhook, or starting a Batch Run. Save with `itops.actions.create` or `itops.actions.save`.

When none are bound to the Site, `itops.automations.emptyHint` uses the shared compact empty-hint treatment and links inline to the New Automation flow.

The Automations list shows `itops.automations.armed` or `itops.automations.disabled`. Use its toggle to arm or disarm a rule, or use the edit and delete actions. Triggered in-app notifications use the Status Bar; popup actions use an app-owned dialog. The Watchdog Status Bar indicator provides app-wide live runtime visibility.

## AI Assistant

The shared AI Assistant receives compact IT Ops page context: Site names/member counts/transports, loaded rack-topology counts, Automation names and armed state, recent-run count, and a live-run summary. It does not receive scripts, host output, secrets, Rack Device details, or full trigger/action bodies through page context. Ask it operational questions normally; for UI guidance it searches this chapter and can navigate to the IT Ops Module or highlight one of the tutorial targets above after you ask to be shown.
