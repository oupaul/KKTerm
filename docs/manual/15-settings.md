## Software updates

Automatic and manual checks normally read `https://kkterm.ryantsai.com/releases/latest.json`. If that app-owned Cloudflare endpoint is unavailable or returns invalid metadata, KKTerm falls back to GitHub Releases. On Windows, installer and checksum downloads also retry the same release asset on the other trusted host if the selected host cannot complete the fetch. A successful check records its time; automatic startup checks are limited to once per 24 hours, while `settings.checkForUpdates` always performs a fresh manual check.

Update checks are not telemetry. They do not upload Connection, Session, Tab, terminal, credential, or usage data. Windows downloads remain protected by the published SHA-256 checksum, and macOS/Linux updates retain Tauri signature verification regardless of whether Cloudflare or GitHub serves the files.

> Settings page styling is consistent across sections. Related controls live inside the shared `settings-subsection settings-fieldset` group, and subsection titles use the shared 12px bottom margin before the normal layout gap. Keep that spacing on the shared title/legend style rather than the first control. Editable controls look editable; disabled / readonly controls stay muted. Delete buttons inside Settings are icon-only red trash cans (no visible "Delete" text). Destructive Settings-wide actions live in **General → Settings data**, behind app-owned confirmation dialogs — not inside feature-specific sections.
> Related Settings rows should stay grouped together inside the same subsection whenever they describe the same topic. Avoid standalone one-row cards for related options; use one fieldset with consecutive rows so the page reads as a compact option group. Hide platform-specific subsections when the current platform has no available rows.

Settings is owned by `src/modules/settings/SettingsPage.tsx` and opens as an app-owned popup over the current Module. Its blocking backdrop covers and disables the Activity Rail as well as the active Module while Settings is open. Persisted bootstrap (`useBootstrapSettings`) lives in `src/lib/settings.ts`; add new persisted settings there, not via cloned effects in `src/App.tsx`.

The universal AI Assistant panel remains visible on Settings. `src/modules/settings/settingsAssistantContext.ts` publishes the active section, visible control keys, and tutorial targets to the assistant. The Tutorial tool can navigate to the owning Settings section before highlighting known targets after the user accepts a navigation offer.

Platform-specific controls are shown only where the runtime supports them. Windows-only controls such as `settings.autoStartWithWindows`, `settings.minimizeToTray`, `settings.useDirectxScreenCapture`, `settings.rdpWebviewStability`, `settings.sectionInstaller`, and `settings.sectionRdp` are hidden in the macOS and Linux builds.

General includes one `settings.activityRail` group for the built-in rail destinations and control. It shows, hides, and reorders by dragging the Workspace (`settings.sectionWorkspace`), Dashboard (`settings.sectionDashboard`), Install Helper (`settings.sectionInstaller`, Windows only), and IT Ops (`settings.sectionItOps`) Modules. These rows use the same colored identity tiles and glyphs as their Module headers. At least one Module must remain enabled. Don't Sleep (`settings.sectionDontSleep`) can be hidden separately and stays with Settings at the bottom of the rail. The reordered Module sequence is saved with the General settings. Workspace, Dashboard, Install Helper, and Don't Sleep default on; IT Ops defaults off. The connected Connection shortcut toggle remains in Workspace because it controls live/pinned Connection entries rather than a built-in Module. Settings navigation also reuses the matching Module identity tile when a dedicated section exists, currently Workspace, Dashboard, and Install Helper.

## General

- Defaults group `settings.generalDefaults`.
- App Update subsection `settings.softwareUpdates`: shows the installed version with `settings.version`, the last manual update check with `settings.lastCheckedAt` / `settings.lastCheckedNever`, and the manual `settings.checkForUpdates` action. Startup update checks are controlled by `settings.autoUpdateChecks` (hint `settings.autoUpdateChecksHint`).
- Language picker: label `settings.language`. Native names come from the `languages` namespace. Changing the selection marks General dirty and applies only after `settings.save`. See [16-localization.md](16-localization.md).
- The global network proxy moved to its own top-level **Proxy** section (`settings.proxy`); see the [Proxy](#proxy) section below.
- Start with Windows minimized: toggle `settings.autoStartWithWindows` (hint `settings.autoStartWithWindowsHint`). When on, KKTerm registers itself for the current Windows user, launches after sign-in, and starts minimized; if `settings.minimizeToTray` is also on, the launch window hides to the system tray.
- Minimize to tray (Windows only): toggle `settings.minimizeToTray` (hint `settings.minimizeToTrayHint`). When on, the title-bar close button hides the window; when off, it exits. Tray "Exit" (`app.trayExit`) always quits. macOS and Linux do not show this control, and app-window close is not diverted there.
- Performance subsection `settings.performance`: toggle `settings.useDirectxScreenCapture` (hint `settings.useDirectxScreenCaptureHint`). When on, screenshot capture tries DXGI Desktop Duplication first and falls back to GDI when DirectX capture is unavailable or unsupported for the requested region.
- Status Bar subsection `settings.statusBar`: toggle `settings.statusBarVisible` (hint `settings.statusBarVisibleHint`) shows or hides the bottom Status Bar. Toggle `settings.statusBarMonitor` (hint `settings.statusBarMonitorHint`) controls whether the visible Status Bar shows and polls CPU/RAM/Network host metrics. When the Status Bar is hidden or the monitor toggle is off, KKTerm stops host usage polling completely. `settings.statusBarMonitorInterval` chooses the polling interval: `settings.statusBarMonitorInterval5`, `settings.statusBarMonitorInterval15`, `settings.statusBarMonitorInterval30`, `settings.statusBarMonitorInterval60`, or `settings.statusBarMonitorInterval300`.
- Settings data subsection (destructive actions live here):
  - Export: `settings.exportSettings` opens the selective export dialog with a switch per category (`settings.segment_connections`, `settings.segment_workspaces`, `settings.segment_dashboards`, `settings.segment_settings`, `settings.segment_mcpServers`) and an off-by-default `settings.includeCredentials` switch that, when enabled, requires a passphrase (`settings.exportPassphrase`) and warns via `settings.includeCredentialsWarning`. Produces a `.kkbackup` file; success `settings.selectiveExportComplete`.
  - Import: `settings.importSettings` opens the backup import dialog. Selecting a `.kkbackup` inspects the bundle and shows a per-category action selector — `settings.importActionSkip` / `settings.importActionAdd` / `settings.importActionReplace` — plus a passphrase field for encrypted bundles. Selecting a full `.zip` backup shows `settings.fullBackupImport` and restores the complete settings database through `settings.importSettingsComplete`, then reloads the app. See [17-data-backup-secrets.md](17-data-backup-secrets.md).
  - Reset all: `settings.resetAllSettings`, confirmation `settings.resetAllSettingsConfirm`, success `settings.resetAllSettingsComplete`.
- Debug subsection `settings.debug`: toggle `settings.advancedDebugging` (hint `settings.advancedDebuggingHint`) enables full AI Assistant, MCP, Install Helper, UI, URL Connection, RDP, SSH, SFTP, Telnet, and heartbeat debug log writing even in release builds. `settings.openLogFolder` opens the local log directory. KKTerm resolves that directory from the running `kkterm.exe` folder first and falls back to `%LOCALAPPDATA%\KKTerm\Logs` if the executable folder cannot be resolved. Enabling advanced debugging writes an `advanced_debugging.enabled` marker to the JSONL logs `aiassistant.debug.log`, `mcp.debug.log`, `installer.helper.debug.log`, `ui.debug.log`, `url.connection.debug.log`, `rdp.debug.log`, `ssh.debug.log`, `sftp.debug.log`, and `telnet.debug.log` beside `kkterm.log`, so the release logging path is visible before the next subsystem event. It also starts `kkterm-heartbeat.debug.log`; turning the setting off stops release heartbeat writes. The `ui.debug.log` records frontend UI diagnostics such as terminal focus-restore attempts (document focus state and the active element) used to troubleshoot input focus after switching apps. The `url.connection.debug.log` records URL Connection Session lifecycle, frontend DOM bounds, clipping, requested overlay bounds, native coordinate conversion, and WebView2 overlay window/client rectangles. The `rdp.debug.log` records RDP startup, ActiveX control creation, display-size sync, and main-thread command timing with non-secret Connection details, and defensively redacts password-like, secret-like, token-like, and credential-like fields. The `ssh.debug.log` records native SSH lifecycle, SOCKS proxy routing metadata with credentials stripped, keepalive configuration, channel open/close events, and terminal input/output byte counts without terminal contents. The `sftp.debug.log` records SFTP browser startup stages, per-session operation waits, transfer start/completion/error summaries, upload write/shutdown timeouts, and partial-upload cleanup without file contents or credential values. The `telnet.debug.log` records Telnet lifecycle, option names and codes, negotiation decisions, terminal-type and window-size events, errors, and byte counts without credential values or terminal contents. These local logs may include prompts, tool arguments, MCP arguments/results, Install Helper command output, URL hostnames, remote hostnames, usernames, proxy endpoints, local paths, remote paths, frontend/native liveness timing, screenshots, and generated Dashboard AI Created Widget source.
- RDP session stability (WebView2): toggle `settings.rdpWebviewStability` (hint `settings.rdpWebviewStabilityHint`) lives in the Debug subsection. When enabled, the next launch applies matching WebView2 stability browser arguments to the main app WebView and URL overlay WebViews while KKTerm runs inside an RDP host.

> Automatic database backups do **not** run from app-window close. Startup backups remain full ZIP snapshots; Settings Export creates `.kkbackup`, and Settings Import accepts `.kkbackup` or full backup `.zip`.

## Credentials & MCP

This is the central manager for secrets stored in the selected credential backend.

- Section header `settings.sectionCredentials`. Stored credentials list `settings.credentialsTitle` / `settings.credentialsStored` (hint `settings.credentialsHint`, empty `settings.credentialsEmpty`).
- Secret storage selector: `settings.credentialStorage` / `settings.credentialStorageBackend`. Windows and macOS default to `settings.credentialStorageOs` and can optionally switch to `settings.credentialStorageFile`; Linux exposes only `settings.credentialStorageFile`. The Credentials page shows a backend-specific security reminder: OS keystore storage uses the operating system credential manager and signed-in account, while encrypted database storage keeps encrypted rows in SQLite and depends on the master password plus database-file protection. Selecting `settings.credentialStorageFile` opens the setup dialog `settings.encryptedSecretStoreSetupTitle`, which repeats the security tradeoff before password entry. If no encrypted SQLite secret store exists, the dialog creates a master password and reminds the user to keep it safe because KKTerm cannot recover it. If one already exists, the dialog verifies the entered master password and unlocks the store. Linux users see a reminder that encrypted database storage is the available backend in this build and should protect the KKTerm database, backups, and `KKTERM_SECRET_STORE_PASSWORD` carefully. Encrypted database storage stays locked until the first feature that needs to read a saved secret, such as opening a Connection with a saved password, or until the user activates the Status Bar `app.credentialStoreUnlockAction`. The unlock dialog includes `settings.encryptedSecretStoreEnvironmentHint` to explain that setting `KKTERM_SECRET_STORE_PASSWORD` before launch lets KKTerm try to unlock the encrypted SQLite store automatically; if verification succeeds, the user does not need to type the master password for that app launch. Manually locking from the Status Bar clears the in-process unlock state, and manually unlocking from the Status Bar always shows the password dialog again. If decryption fails, the dialog can create a new master password, which invalidates and clears encrypted credentials saved with the previous password. The switch changes which backend KKTerm reads and writes from. It does not migrate existing secrets between stores.

## AI Assistant

Section header `settings.sectionAiAssistant`. Owned by `src/modules/settings/AiSettings.tsx`. Per-provider configuration lives in `src/ai/providerRegistry/`.

- Provider picker; known-model picker is a real `<select>` showing every model — not an `<input list>`/`datalist` (Chromium hides non-matching options behind a `datalist`).
- GitHub Copilot model refresh uses the signed-in account's Copilot CLI model catalog. The `settings.navigation.auto` model option leaves model choice to the Copilot runtime instead of sending a fixed model id; after a successful refresh, the picker shows `settings.navigation.auto` plus curated Copilot models that are also available to the signed-in account.
- Custom model ID is a separate text input.
- OpenAI Compatible providers can choose API request mode with `settings.apiMode`: `settings.apiModeChatCompletions` uses `/chat/completions`, and `settings.apiModeResponses` uses `/responses`.
- OpenAI Compatible providers can set `settings.extraHeaders` as comma-separated `key=value` pairs; example placeholder `settings.extraHeadersPlaceholder`. These headers are provider request metadata, not credential-backend secrets.
- API keys go into the selected credential backend under `AI_PROVIDER_SECRET_OWNER_ID`; never written to SQLite or settings JSON.
- Local Ollama uses the local server endpoint and does not show an API key field. Ollama Cloud is the Ollama provider that stores an API key.
- When a secret write to the credential backend fails (for example macOS Keychain access was denied), the save is reported as `settings.secretSaveFailed` and aborts before the provider settings persist, so the active provider does not change. Saving then opens the `settings.keychainSaveFailedTitle` recovery dialog (`ConfirmSheet`, `warn` tone) with body hint `settings.keychainResetHint`; its `settings.keychainResetAndRetry` action deletes the stored entries the draft is about to write and re-runs the save so the OS re-prompts for access. There is no deep link to a macOS keychain permission pane because keychain access is governed by the item ACL in Keychain Access.app, not by a System Settings privacy category.
- OpenAI exposes `settings.useCodexCli`; Anthropic exposes `settings.useClaudeCli`. When enabled, the API key entry is disabled and KKTerm delegates AI Assistant turns to a local CLI backend instead of the HTTP API-key provider path. KKTerm prefers the matching Agent Client Protocol stdio adapter, attaches the built-in `kkterm` MCP server for published safe tools such as Connection creation, then falls back to the vendor CLI's documented one-shot command mode if ACP is unavailable. The one-shot fallback is suggest-only for KKTerm tools. The setup row can check CLI install/auth status, start the matching Install Helper recipe (`codex-cli` or `claude-code-cli`) with missing prerequisites, and open the vendor auth command in an external terminal.
- CLI install/auth status checks are manual-only. The setup row persists the last manual result and shows its `settings.lastCheckedAt` timestamp when the user leaves and returns to Settings.
- Tool permission default (`ai.toolPermissionMode`) is set here as well.
- Assistant tools group: title `settings.aiToolsTitle`. It is collapsed by default; expand/collapse uses `common.expand` / `common.collapse`. Assistant tools default enabled, including `settings.aiTools.network.label` and `settings.aiTools.memory.label`, except `settings.aiTools.email.label`, which stays off until the user enables and configures delivery. The Tutorial tool is `settings.aiTools.tutorial.label` / `settings.aiTools.tutorial.description`. The Assistant memory tool (`settings.aiTools.memory.label` / `settings.aiTools.memory.description`) lets the assistant save and recall short durable notes about the user's hosts and preferences across chats; secrets are never stored as notes.
- Assistant Skills subgroup: title `settings.assistantSkillsTitle` (hint `settings.assistantSkillsHint`, empty `settings.assistantSkillsEmpty`). It is collapsed by default; expand/collapse uses `common.expand` / `common.collapse`. KKTerm copies missing bundled starter skills into the local app-data skills folder: `dashboard-widget-builder`, `dashboard-widget-designer`, `dashboard-data-visualization`, `desktop-accessibility-ui`, `dns-dhcp-troubleshooter`, `firewall-port-troubleshooter`, `network-connectivity-troubleshooter`, `remote-desktop-helper`, `sftp-transfer-helper`, `ssh-troubleshooter`, `terminal-command-planner`, and `tls-certificate-troubleshooter`. `settings.assistantSkillsOpenFolder` opens that folder where users add or edit one SKILL.md-compatible directory per skill. The Custom Skills row (`settings.assistantCustomSkillsTitle`, hint `settings.assistantCustomSkillsHint`) controls whether extra skill directories under `assistant-skills\custom` are included when the Assistant Skill list is refreshed and when the AI Assistant starts a request; `settings.assistantCustomSkillsOpenFolder` opens that folder. Per-row actions open that skill directory (`settings.assistantSkillsOpen`) and toggle whether the assistant can invoke it (`settings.assistantSkillsEnabled` / `settings.assistantSkillsDisabled`). Skill selection is model-driven through `assistant_use_skill`, not app keyword matching. v1 does not execute skill scripts.

## SSH

The SSH key generator accepts an optional passphrase plus confirmation (`settings.sshKeyPassphraseOptional`, `settings.sshKeyPassphraseConfirm`) and passes it to `ssh-keygen`; leaving it blank generates an unencrypted key.

Section header `settings.sectionSsh`. Default username, default identity file, agent forwarding, tmux defaults, etc. (Keep this section keyed under `settings.*` — exact field keys live in `en.json`.) The `connections.useTmux` row in Settings - SSH controls whether Add Connection defaults SSH Connections to tmux management; the per-Connection override lives in the add/edit Connection dialog's right-column SSH options. `settings.proxyJump` defines the default jump host for new SSH Connections. The SSH SOCKS proxy default moved to Settings → Proxy (`settings.proxy`): an SSH Connection that inherits defaults and has no per-Connection SOCKS override falls back to the global app proxy when it is a manual SOCKS5 endpoint (terminals, tmux, SFTP, and key transfer), dialed natively in-process with no external tools and never combined with ProxyJump. Per-Connection SSH SOCKS overrides remain in the add/edit Connection dialog under `connections.sshProxyOptions`, including `connections.sshSocksProxyUsernameOptional` and `connections.sshSocksProxyPasswordOptional` when Settings defaults are not inherited; inline `username:password@host:port` is accepted in the per-Connection address field. When `connections.inheritSettingsDefaults` is on in that dialog, ProxyJump and `connections.useTmux` show the current Settings defaults while the SOCKS fields stay blank so the launch uses the global app proxy; saving writes those displayed non-secret values into the saved Connection. Turning `connections.inheritSettingsDefaults` off makes those same fields editable and saves explicit per-Connection values; the per-Connection SOCKS proxy password is stored under that Connection's secret owner id rather than in SQLite. `settings.defaultTransparency` sets the starting SSH terminal transparency for new SSH Connections and Child Connection Tabs; the default is 50. `settings.randomDynamicBackgroundOnCreate` assigns a random dynamic terminal background only when creating new SSH Connections, new top-strip Tabs from SSH Connections, or new SSH Child Connection Tabs. SSH port forwarding mappings are edited from the SSH Pane dialog, not Settings.

The `settings.xServer` group controls the managed VcXsrv launcher. `settings.xServerManaged` starts VcXsrv before opening SSH Sessions when `vcxsrv.exe` is not already running and shows the Status Bar X indicator while the setting is enabled. The indicator reflects configuration, not command-based process polling. `settings.xServerPath` can override the executable path; blank uses standard VcXsrv install locations. `settings.xServerDisplay` chooses the local X display number, and `settings.xServerArgs` passes command-line flags. `settings.xServerLaunch` saves the current SSH Settings draft and starts VcXsrv immediately.

X11 forwarding is negotiated when a new native SSH Session starts. SSH Sessions that were already open before enabling or restarting VcXsrv need to be reconnected or opened again before remote X11 apps receive `DISPLAY`. Existing tmux panes can also keep an old shell environment; open a new tmux pane/window or export the new `DISPLAY` inside that shell.

## Terminal

Section header `settings.sectionTerminal`. Font family + size, line height, cursor style, scrollback length, bell behaviour, default shell on Local, and custom local shell profiles. Clipboard option `settings.copyOnSelect` copies a completed terminal selection through the same system-clipboard path as the explicit Copy command, for local and remote Sessions alike; toggling it takes effect immediately in already-open terminal Sessions. Its description `settings.copyOnSelectHint` explains that ordinary SSH selection copies immediately, tmux mouse mode requires Shift-selection (or disabling mouse mode), and tmux copy-mode selections do not trigger this option. The font family field (`settings.terminalFontFamily`) is grouped as Custom fonts, Recommended, then System fonts: user-dropped custom fonts (`settings.customFonts`) stay first, the platform-specific monospace list led by `settings.terminalFontDefault` follows inside `settings.recommendedFonts`, and OS-detected fonts (`settings.systemFonts`) come last. The default terminal font is a single CSS stack that resolves to each platform's native monospace via font fallback: Cascadia Mono on Windows, SF Mono on macOS, and the bundled JetBrains Mono on Linux (or anywhere the others are absent), so `settings.terminalFontDefault` names whichever the platform resolves. JetBrains Mono ships bundled with KKTerm and is therefore always offered as a Recommended terminal font on every platform, guaranteeing the stack renders even with no system monospace installed. macOS recommendations also include SF Mono and Menlo; Linux recommendations include Adwaita Mono, Ubuntu Mono, and common cross-distribution coding fonts; Windows retains Cascadia and Consolas choices. The refresh icon (`settings.refreshSystemFonts`) uses the same asynchronous shared refresh as Appearance, so activating either button rescans custom and system fonts, updates both pickers, and filters recommendations against the detected families without blocking initial render (status `settings.systemFontsRefreshed`, unavailable runtime `settings.systemFontsUnavailable`). Custom fonts are still managed from the app fonts folder, the shared hint is `settings.customFontsHint`, and the open-folder action is the icon-only `settings.openCustomFontsFolder` button. TTF/OTF files with the same internal family appear as one Custom fonts entry and contribute their available Regular, Bold, and Italic faces; missing faces do not prevent using the family. Nerd Font Normal, Mono, and Propo variants remain separate entries. App UI may use any variant, but Terminal restricts both the Custom fonts and System fonts groups to families whose metadata reports fixed-width (monospace) cells, so proportional faces never appear in the Terminal picker; an older saved proportional custom terminal family resets to the terminal default without changing other Terminal settings. The selected terminal font applies immediately to mounted local and SSH terminal Sessions: KKTerm refreshes xterm glyph metrics and refits the Pane without reconnecting. Custom fonts — including powerline/Nerd Fonts — render without a system-wide install because KKTerm loads them into the WebView itself; when asynchronous custom faces finish loading, mounted terminal Panes refresh again so fallback glyph metrics are not retained. The refreshed scanner prefers English metadata names so localized aliases in System fonts do not duplicate Recommended entries. The Custom shells subsection (`settings.customShells`) stores named command lines with optional arguments; the name field suggests common platform shells and fills a recommended command line that remains editable for non-default install paths. Saved entries appear in `settings.defaultShell` and in Add Connection → Local terminal → `connections.shell`. `settings.defaultTransparency` sets the starting local/Telnet/Serial terminal transparency for new local/Telnet/Serial terminal Connections and Child Connection Tabs; the default is 50. `settings.randomDynamicBackgroundOnCreate` assigns a random dynamic terminal background only when creating new local/Telnet/Serial Connections, new top-strip Tabs, or new Child Connection Tabs.

## RDP and VNC

- `settings.sectionRdp` — RDP defaults: `settings.remoteDesktopViewMode`, resolution, colour depth, redirection toggles.
- `settings.sectionVnc` — VNC defaults: `settings.remoteDesktopViewMode`, colour depth, view-only, encoding preferences. The view-mode default controls how new or inheriting VNC Connections display large or multi-monitor framebuffers before a per-Connection override is selected from the remote desktop toolbar.

## URL

`settings.sectionUrl` — defaults for URL Connections. The proxy default moved to Settings → Proxy (`settings.proxy`); URL Tabs that inherit defaults follow that global app proxy, and a per-Connection override still wins. `settings.urlDataShards` manages URL data shard labels and includes the global default shard used by URL Connections that inherit Settings defaults. `settings.urlUserAgent` sets an optional global default user-agent string for URL Connections; blank keeps the platform webview default, and individual URL Connections can override it in `connections.urlOptions`. `settings.urlSavedPasswords` can show multiple saved password/input-data entries for the same URL Connection when a site uses multi-page authentication; each entry is keyed by the page origin and path, with temporary query strings and fragments ignored. Each record keeps its Connection name, truncated page URL, Edit, and Delete controls on one row. Edit opens the shared compact editor dialog instead of expanding an inline form. Each editor row shows the captured field's exact selector and matching-element occurrence beside its value. The saved primary password starts masked and has an eye control; non-password values start visible and have a lock control that persists whether the editor masks them next time, plus an eye control for temporary reveal while masked. Masking a non-password value changes only its editor presentation and does not move that SQLite value into the credential backend. Settings → Credentials renders the same records and controls through the same component. A URL Connection may inherit these defaults, force direct access, or store its own endpoint and shard from the add/edit Connection dialog's right-column `connections.urlOptions`. HTTP mode carries both HTTP and HTTPS destinations; authenticated proxies, PAC files, and bypass lists are outside Tauri's per-WebView proxy API. macOS requires version 14 or later. URL Connections use stable owned overlay windows, not Tauri's `unstable` child-webview API. The `unstable` child-webview path was removed because it changed the main app WebView2 host shape and could break terminal keyboard input after Alt+Tab/app switch or minimize/restore until the user clicked the terminal Pane again.

## Proxy

Section header `settings.proxy` (rendered without a section-header glyph, following the Settings page convention). This is a top-level Settings section in the left sidebar, listed above About, and is reached with `tutorial_highlight` target `settings.proxy` (navigation `page=settings settingsSectionId=proxy-settings`).

- Subsection `settings.proxy` (hint `settings.proxyHint`): the global network proxy for all KKTerm activity — connections, app updates, AI providers, and other web requests. `settings.proxyMode` selects `settings.proxyModeSystem` (follow the operating system proxy: Windows WinINET, macOS CFNetwork, Linux environment variables), `settings.proxyModeNone` (force a direct connection), or `settings.proxyModeManual`. Manual mode shows `settings.proxyProtocol` (`settings.proxyHttp`, `settings.proxyHttps`, `settings.proxySocks5`), `settings.proxyHost` (placeholder `settings.proxyHostPlaceholder`), and `settings.proxyPort`. Per `settings.proxyPlatformHint`, a manual SOCKS5 proxy also routes SSH/SFTP and Telnet sessions, while HTTP/HTTPS proxies apply to web and app requests only. The proxy applies on next request/Session start; URL Tabs and reqwest activities pick it up when they next open. Authenticated proxies, PAC files, and bypass lists are not configured here.
- Save status: `settings.proxySettingsSaved`. The proxy mode and manual endpoint persist into the General settings record (`proxyMode` / `proxyUrl`) even though the controls now live in this dedicated section.

## About

`settings.sectionAbout`. Shows version (`settings.version`) and slogan (`settings.appSlogan`). The version value should match `package.json`'s `version` field. License info, GitHub link, and acknowledgements live here.

Manual app update checks live in General -> `settings.softwareUpdates`, not About. `settings.autoUpdateChecks` controls startup update checks, and manual checks surface update state through `settings.checkingForUpdates`, `settings.updateNoUpdates`, and `settings.updateCheckFailed`. When an app update is available, the `settings.updatePromptLabel` dialog shows `settings.updateAvailableBody`, renders `settings.updateNotes` from the release markdown or updater metadata, and offers `settings.updateOpenDownloadPage`, `settings.updateDownloadAndInstall`, or `settings.updateLater` when an installable update is available for the current platform. Download and install replaces the prompt with the shared determinate Status Bar popup, driven by actual downloaded bytes. On Windows, its close button uses `settings.updateDownloadCancel` to stop the active transfer and delete the partial installer; after checksum verification, the popup stays at 100% for three seconds, then KKTerm exits and launches the installer. When the installer exits successfully, the detached handoff deletes that downloaded installer and removes the empty update directory; a failed or cancelled installer is retained. On macOS and Linux, the signed Tauri updater reports real progress and follows the same three-second completion delay before installation and relaunch; the current updater API does not expose reliable in-flight cancellation, so it does not show a cancel control while downloading.


## Built-in MCP Server

- Location: Settings → AI Assistant section on Windows, macOS, and Linux. The bridge transport is a named pipe on Windows and a `0600` Unix domain socket on macOS/Linux; the built-in MCP rows are shown on all three.
- Toggle keys: `settings.builtInMcpServerEnabled` (`settings.builtInMcpServerEnabledHint`) and `settings.builtInMcpAllowAllDangerous` (`settings.builtInMcpAllowAllDangerousHint`).
- Config dialog keys: `settings.builtInMcpShowConfig`, `settings.builtInMcpConfigTitle`, `settings.builtInMcpConfigIntro`, `settings.builtInMcpConfigFormatJson`, `settings.builtInMcpConfigFormatToml`, `settings.builtInMcpConfigCopy`, `settings.builtInMcpConfigCopied`, `settings.builtInMcpConfigLocationsTitle`, `settings.builtInMcpConfig*Header`, and `settings.builtInMcpConfigMethod*`.
- Config dialog behavior: opens only when the user activates `settings.builtInMcpShowConfig`; changing either MCP toggle never opens the dialog. It shows copyable JSON (`mcpServers.kkterm`) and TOML (`[mcp_servers.kkterm]`) snippets for stdio MCP clients using the resolved `kkterm-cli` path beside the running KKTerm executable (`kkterm-cli.exe` on Windows). The setup table shows localized Agent/Method/Project/Global headings. Codex and Claude Code rows include documented CLI commands where supported; VS Code/GitHub Copilot, Antigravity, and OpenCode rows use `settings.builtInMcpConfigMethodManualEdit` and list config locations.
- Purpose: enable/disable the local built-in MCP server surface and control whether built-in MCP tools with a `dangerous` namespace segment, such as `kkterm.workspace.quick_commands.dangerous.create`, require confirmation prompts or run in allow-all mode.
- Debug logging: debug builds write built-in and remote MCP request/response records to `mcp.debug.log` beside `kkterm.log`; release builds write the same MCP log when `settings.advancedDebugging` is enabled. Built-in MCP logging redacts terminal send input, terminal buffer reads, Dashboard widget source/body JSON, and secret-looking argument fields before writing debug records.
